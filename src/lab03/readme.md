**Тема:** Наследование, иерархия классов и чистокровный полиморфизм без костылей на Python.


## 1. Цель работы
Освоить механизм наследования классов, научиться строить логичную иерархию объектов и безжалостно переиспользовать код. Понять разницу между базовым и производным классами, победить дублирование с помощью `super()` и освоить полиморфное поведение.


## 2. Ключевые темы ЛР-3

### 1. Базовый класс (Фундамент банка)
Класс `Client` (из ЛР-1) теперь выступает в роли железного предка для всех типов аккаунтов. Он содержит общие для всех клиентов атрибуты (имя, базовый баланс) и стандартные методы отображения.

### 2. Производные классы (Специализация по выкачиванию денег)
На основе базового класса построена иерархия из двух дочерних классов, расширяющих возможности системы:
* **`VIPClient` (Элита с бонусами):** получает 2 новых атрибута (персональный менеджер и процент кэшбека) и уникальный метод `.gift_premium_status()`, радующий богачей.
* **`DebtorClient` (Злостные неплательщики):** обзавелся атрибутами (размер долга по кредитке и количество дней просрочки) и карательным методом `.send_collectors()`, отправляющим коллекторов к должнику.

При инициализации жестко используется `super()`, чтобы не дублировать код конструктора родителя.

### 3. Переопределение методов и полиморфизм без условий
* **Красивый вывод (`__str__`):** каждый подкласс переопределяет стандартный вывод, чтобы в консоли VIP-клиенты сверкали статусом, а должники сразу подсвечивались своими штрафами.
* **Полиморфизм без условных операторов:** введен единый интерфейс — метод `.calculate_annual_maintenance()`. Никаких анти-паттернов вроде `if type == ...`! Метод вызывается одинаково для всех, но VIP-клиенту он считает скидку, а должнику накидывает штрафной тариф.

### 4. Интеграция с коллекцией
Банковское хранилище (`ClientCollection` из ЛР-2) адаптировано для одновременного хранения и VIP-персон, и должников. Коллекция умеет корректно работать с ними через один цикл, а также реализует метод `get_only_debtors()` для быстрой фильтрации и выборки конкретного типа клиентов с помощью `isinstance()`.

В ходе работы архитектура «Золотого Насоса» была переведена на рельсы ООП-наследования. Использование `super()` избавило код от дублирования, а внедрение полиморфизма позволило обрабатывать разношерстных клиентов внутри одной коллекции без костыльных проверок типов.
